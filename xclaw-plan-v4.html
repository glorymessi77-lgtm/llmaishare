01 · CORE INSIGHT

核心洞察

同一套 XClaw 能力栈支撑两个教学场景：教师 AI 助手帮老师跑工作流；教师数字分身陪伴学生答疑。教师创编的 Skill 和智能体——一份资产，两种用法。

03 · USER PAINS

师生用户痛点

教师场景 4 个工作流断点 + 学生场景 5 个陪伴断点。两侧痛点不是各自独立，而是用同一套教师创编资产闭环解决。

教师场景 · 工作流断点

4 痛

备建课断点

高 · 每周 3-5 次
找完资料组合成完整一节课要 1 小时体力活 · 新课建课需要重新搭大纲/章节/资源

教学断点

高 · 每天
班级 60 人顾不过来薄弱学生 · 课后跟进难 · 分身需要持续调教保持"像我"风格 · 30 年方法论无法数字化沉淀

考评断点

高 · 月度峰值
批改完只完成 50%——错题归因/学情归档没时间做，但每次评估都被问

督管断点

高 · 每周高频
督导员一周写 10 份 800 字结构化报告，是重复劳动最严重的岗位

学生场景 · 自主学习断点

4 痛

答疑断点

高 · 每天
老师下课就找不到了 · 深夜难题没人问 · 通用 AI 答的不是"我老师的讲法"

学习路径断点

高 · 每周
不知道下一步该学什么 · 全班齐步走的进度 ≠ 适合我的节奏 · 需结合本课程进度+我的学情

学习资源断点

高 · 每次自学
现有课件/习题不适配我的水平 · 找不到"刚好适合我"的解释方式与练习 · OpenMAIC 交互式课件生成已客户验证

学情自检断点

高 · 每次考后
考完不知道自己哪错了、哪类知识点弱 · 老师顾不过来逐个反馈 · 学生需要自己看得见自己的弱点

05 · TOPIC ① · SKILL

专题① Skill 库

Skill 是 XClaw 里所有"动作能力"的统一形态——无人格、无状态、可复用。按颗粒度分原子 Skill 和工作流 Skill 两种；按归属分私人、课程、学校三层共享。

原子 SKILL

单一动作单元

做一件具体的事 · 输入 → 输出 · 一次调用即完成 关键特征 可独立测试 / 可被多个工作流复用 / 易于教学专家审核
出题 批改 评语生成 错题归因 知识点解释 板书要点 课堂讨论问题 学情周报
工作流 SKILL

多 Skill 串联流程

端到端跑通一个完整教学场景 · 内部调度多个原子 Skill + MCP 关键特征 解决断点 / 串起多系统 / 输出"可直接用的成品"
备课流程 考评流程 督管流程 学情归档 跨学期复盘
关键架构原则 · 不要做"超级 Skill"
内部反馈："自主学习 Skill 开发难度极大"——因为把答疑+学习路径+学情+资源生成都打包进一个 Skill 里。

正确做法：拆成 4 个原子 Skill + 1 个工作流 Skill 由智能体调度。架构上"自主学习"不是一个 Skill，而是教师数字分身（智能体）调度 4 个原子 Skill呈现的场景体验。
❌ 一个"自主学习 Skill" = 答疑 + 路径 + 学情 + 资源（难写、难维护、难复用）
✅ 4 原子 Skill：答疑 / 路径推荐 / 学情归因 / 课件生成
    + 教师数字分身串联调度 → 学生感知"自主学习场景"
跨场景共享的原子 Skill（同一 Skill 被多场景调用）

同一个底层 Skill 在不同场景中被不同智能体调用——这才是"原子 Skill"分层的真正价值

06 · TOPIC ② · AGENT

专题② 智能体

智能体 ≠ Skill。智能体是"持续角色"——有人格、有记忆、能多轮陪伴。XClaw 两个核心智能体：教师 AI 助手（教师场景）+ 教师数字分身（学生场景）。

维度 🔧 Skill 🎭 智能体
本质工具 · 动作角色 · 人格
人格无（按指令执行）有（教师语气/节奏/风格）
记忆无状态（每次独立）持久记忆（懂这门课、懂这个学生）
交互单次输入 → 单次输出多轮对话 · 陪伴式
颗粒度小（一 Skill 一动作）大（一智能体 = 一个角色）
关系智能体的"工具包"Skill 包 + 人格 + 记忆 + 上下文
类比厨师的菜谱厨师本人的数字版
两个核心智能体（XClaw 26 年聚焦）
教师场景

① 教师 AI 助手

教师每天打开的工作伙伴，调度自己创编的 Skill 库
教师 AI 助手 = 教师 Skill 库 + 工作区记忆 + 对话上下文
  • 主入口形态聊天框 + Skill 工具箱（类 OpenClaw 体验）
  • 核心能力调度备建课、教学、考评、督管四大工作流 Skill
  • 人格记忆记住老师工作习惯、常用 Skill、教学风格
  • 跨学期记忆"去年这门课你怎么讲的、学情如何"
  • 未来形态教师可选择不同"工作模式"（备课模式/批改模式/督管模式）
学生场景

② 教师数字分身

学生 24h 陪伴的"老师数字版"——风格、内容、上下文都还原
教师分身 = 答疑风格 + 课程内容深度记忆 + 学生学情上下文
  • 学生感知"本课程老师的分身"——不是通用 AI 是这门课的老师

09 · KEY CLARIFICATIONS

5 个产品定型关键澄清

产品定型阶段的 5 个关键命题——概念边界、技术选型、场景范围、架构层级。下面是每条的产品事实陈述。

1 Skill 是概念家族，不是 OpenClaw 专属 概念边界
Skill 分三层来看：
  • ① Skill 概念（可复用动作单元）= 通用 · 所有智能体平台都有等价物（Coze 插件 / Dify 工具 / ima.copilot Skill）
  • ② OpenClaw 原生 Skill 格式（markdown + 工具调用 + 沙箱）= 龙虾系特有
  • ③ 能调度 Skill 的产品 = 大概率是 Claude 系生态二次封装（CC / Cursor / Codex / ima.copilot 等）
对外口径：XClaw 选 OpenClaw 内核是为了拿最强 Skill 调度能力，但 Skill 概念不被"龙虾"绑死。
2 学生不接触 XClaw 本身，技术层由 XClaw Engine 调度 产品定义
技术实现 vs 用户感知是两件事：
  • 产品层（用户感知）：学生用的是"教师数字分身"产品形态，不需要懂 XClaw
  • 技术层（运行时）：分身背后由 XClaw Engine 调度教师创编的 Skill
类比：用户用微信，不需要懂腾讯云。学生感知"伴学空间 / 数字助教"，背后是 XClaw Engine。

① MCP
Model Control Protocol · 标准化协议
是什么
一套标准化协议，把平台所有能力包装成统一服务，任何智能体都能调用
✓ 优点
  • 生态开放，可接 OpenClaw / QCloud / WorkBot 等
  • 能力全，一次包装到处可用
  • 适合长期战略、平台化
✗ 缺点
  • Token 消耗极大（每次调用读全量信息）
  • 重、慢、成本高
  • 校内私有化场景没必要
⊙ 适用
  • 要做开放生态、允许外部智能体接入
  • 多厂商、多产品互通
  • 学校未来要接入市面主流 AI 产品
② CLI
Command Line · 命令行调用
是什么
轻量化指令调用，直接通过命令触发平台接口，最轻最快
✓ 优点
  • 极轻、无冗余、速度快
  • 开发成本低、稳定
  • 自家产品打通最优解
✗ 缺点
  • 不开放、不通用
  • 不适合给外部智能体用
  • 无标准化生态
⊙ 适用
  • 内部产品打通（XClaw + 教学平台）
  • 不需要对外开放，只给自己用
  • 追求低成本、高性能、私有化部署
③ Skill
技能 / 工作流封装
是什么
把业务动作封装成可复用流程，比如：自动备课、生成课件、批改作业
✓ 优点
  • 面向教学业务，老师能理解、能用
  • 可编排、可沉淀、可共享
  • 天然适配教学场景
✗ 缺点
  • 偏业务，不负责底层协议
  • 复杂联动仍需依赖接口 / MCP
⊙ 适用
  • 教师自动化教学
  • 备讲课、答疑、考评、督学
  • 技能广场、资产复用
  • 给老师 / 学生用的上层功能

  • ❌ 不在 XClaw 发力：评（评教半年一次价值低）+ 管（教务/学籍/管理层在平台体现不出来）
Q3/Q4 资源聚焦四大教师场景 + 自主学习。
10 · STRESS TEST

对抗性自检

这套规划可能错在哪——主动暴露风险，欢迎教学专家与跨团队挑战。

1"教师创编 → 学生分身"链路真的成立吗？
风险：教师创编的 Skill 是"出题方法"，不代表他想以这种风格"答疑"——风格漂移会让学生觉得"这不是我老师"。
补救：教师分身必须在 Skill 之上加一层"答疑场景适配"（语气/节奏/边界）。这一层算法工作量已纳入研究院 Q4 排期。
2Skill 和工作流合并到一层会不会混乱？
风险：原子 Skill 和工作流 Skill 颗粒度差异大 · 在 UI 上混在一起，教师/教学专家可能找不到入口。
补救：架构上一层，但 Skill 库 UI 上用标签/分类清晰区分"原子 / 工作流" · 教学专家审核时按颗粒度分别管理。
3Vibecoding 教师真的会用吗？
风险：80% 教师可能不会主动创编 → 创编机制再好也变成摆设。
补救：把创编入口放主路径——每次 AI 调用后强引导"要不要把这个动作存成 Skill" · 课程组长牵头共建 · 共享激励纳入年度教学创新成果。
4"6 个 Skill + 13 个 MCP"是不是过度承诺？
风险：教学专家产 Skill 慢、小雅产 MCP 慢——两边一卡 XClaw 就空转，Q3 兑现不了。
补救：P0 五件套必须独立可验证——浏览器自动化 + 教研室手工 1-2 个 Skill 也要能跑出 Demo · 不能等所有方齐了才动。
5小雅缺"教师空间"怎么办？
风险：教师 AI 助手部署位置不明 · 内嵌"课程空间"可能体验割裂。
补救：Q3 先以小雅"课程空间"为载体内嵌 · 同步推动小雅产品规划补强"教师空间" · 形成完整三大空间矩阵（教师 / 课程 / 学生）。
6本地化部署 + 跨学期累积 → 磁盘怎么办？
风险：一门课工作区 5 年累积 5-20GB · 10000 门课 × 5 年 = 50-200TB · 校方机房扛不扛得住？
补救：合同必须写清存储扩容条款 · 工作区做冷热数据分层 · 跨年归档压缩到对象存储。
