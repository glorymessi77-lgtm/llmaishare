01 · PM 责任分配总览

OWNERSHIP
两层第一责任人 · 职责定义

Epic 总体负责人(Epic 第一责任人)— 负责该 Epic 内所有用户故事的统一评审组织:

  • 统一评审组织 — 把 Epic 内多个故事打包成一次或几次评审会,而不是每个故事单独评审
  • Epic 节奏把控 — 协调多个故事的 PRD 写作进度 / 评审排期 / 上线节奏
  • 跨故事依赖协调 — 处理 Epic 内多个故事之间的依赖、冲突、资源调度
  • Epic 整体对外汇报 — 老板/客户问该 Epic 进展找他

故事第一责任人(Story 第一责任人)— 负责单个用户故事的具体执行:

  • PRD 主笔人 — 负责该故事 6W 完整描述
  • 验收对接人 — 测试通过后跟客户/上线对接
  • 出问题先找谁 — 团队成员问该故事细节都找他

协同 PM 不是替代,是"该故事内某些 Task 的执行者"。第一责任人要主动协调他们,但责任不分摊。
Epic 总体负责人和故事第一责任人可以是同一人,也可以是不同人(如 E5 总体马鸣珂,U11 故事第一责任人是杨晓丽,马鸣珂仍组织 U10+U11 整体评审)。

Epic 总体负责人速查(7 个 Epic)

每位 Epic 总体负责人负责组织该 Epic 内所有用户故事的统一评审 — 这意味着邱月组织 U04 一次评审 / 杨晓丽组织 U01+U02 一次评审、U06+U07+U08+U09 一次评审 / 张能组织 U03 一次评审、U13+U14+U15+U16 一次评审 / 马鸣珂组织 U10+U11 一次评审 / 徐彤彤组织 U18+U19 一次评审 = 共 7 场 Epic 级评审会。

备注 · 用户故事的目标和原则

PRINCIPLES

这一节是前置方法论备注 — PM 在看下面的 16 个故事卡之前,请先用这个标准对照自己负责的故事,自查 6W 是否完整。后续 V3.0.5 / 新故事的整合也沿用这套方法论。

用户故事的目标 · 标准格式

3 段式标准描述

作为 [角色](教师 / 学生 / 管理员 / 教务处 / 第三方督导)
我希望 [可观察的目标](具体能做什么,不是功能名)
以便 [获得的价值](为什么对我有用)

反例:"AI 任务团子模块功能升级"(功能视角,不是用户视角)
正例:"教师用 AI 组织任务团子,学生通过闯关完成学习,降低教师反复组织作业测练的成本"(用户视角)

6 大整合原则(写 PRD 时自查)

原则核心要求判断口诀
① 用户场景统一 同用户 + 同场景 + 同目标 = 一个故事 "4 个 Task 拆 4 个 PRD 还是合 1 个?"
② 目标驱动 故事标题写"用户目标",不是"功能模块名" "模块升级"=错 / "教师做 X 获得 Y"=对
③ 横切类剥离 跨场景的体验/底座/合规独立成横切 Epic "暗色模式 / 登录 / 多语言 / 部署"分出去
④ 依赖项挂载 底座能力作为独立 Epic,业务故事引用不重写 PRD 里写"依赖 E7 AI 基座",别重复底座描述
⑤ Minor fix 合并 注 XY2.5-xxxxx 的小调整合并到主故事,不单列 "8 个看板 minor fix 全合 U06,不要拆 8 个故事"
⑥ 6W 完整性自查 每个故事必须回答 6 个问题(下面表格) "少一个 W = 故事不完整,要么拆要么合"

6W 标准 PRD 自查表

维度核心问题案例 · U04 任务团子智能化
Who谁用?教师(主)+ 学生(辅)
Why为什么?用 AI 降低出题/批改成本,游戏化激发学生学习
What要什么?AI 生成任务团子内容、反复迭代、闯关、纳入成绩
Where在哪?课程空间 → 作业任务模块
When什么时候?教师备课时建,学生学习时做,期末考核计分
How验收标准?AI 一次生成 ≥3 类内容、支持微调、闯关条件可配置、成绩可纳入总评

5 个 PM 常踩的坑(写 PRD 前对照检查)

坑❌ 错✅ 对
故事拆得过细"AI 任务团子生成" + "AI 任务团子微调" = 2 个故事合并为"任务团子智能化"(同一目标 = 一个故事)
故事合得过粗"教师建课" = 一个故事拆成 U01 资产引用 + U02 图谱深化(能独立体验闭环 = 一个故事)
横切类塞进业务故事"MOOC 学习链路 + 暗色模式" 一个故事暗色独立成 U13(跨场景体验 ≠ 业务故事)
PRD 反复写底座每个故事 PRD 里都重复"AI 基座"描述用"依赖 E7"引用,正文不重复
故事描述用功能名"AI 任务团子模块功能升级""教师用 AI 组织任务团子,学生通过闯关完成学习"(目标视角)
