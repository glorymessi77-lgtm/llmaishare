01 · 为什么需要多智能体

为什么需要多智能体

当你让 AI 先做市场分析，再做竞品调研，最后出产品方案——到第三步时，第一步的结论已经开始模糊了。这不是 Prompt 写得不好，而是单一对话模式的结构性天花板。

以下三个瓶颈，是任何"把所有事情塞进一个对话窗口"的方式都无法回避的根本问题。它们不是靠优化 Prompt 能解决的，而是需要架构层面的突破。

01
上下文越聊越长，前面全忘了
对话越长，AI 越容易丢失早期的关键结论。你精心引导的市场分析结果，到了出方案的时候已经被"挤出"了上下文窗口。上下文窗口是有限资源。
真实案例：跑了两天的会话，上下文接近上限
02
明明能并行，却只能串行
市场分析和竞品调研完全可以同时做，但单一对话只能一个接一个来。本来 30 分钟能做完的评估，2 小时也没搞定。
真实案例：多窗口并行完成任务
03
一个视角，顾此失彼
让 AI 关注市场机会就容易忽略技术风险，关注技术方案就忽略商业价值。没法多维度同时评估。
真实场景
你让 AI 分析"AI能力训练平台"的市场空间，它输出了乐观的 30 亿预估。但同一个 AI 无法同时站在算法总工的角度质疑："这个规模需要的算力成本你算过吗？"——因为它只有一个视角。
⚡ 关键结论
这三个问题的共同根源是：单一对话窗口既是"执行者"又是"记录者"又是"协调者"。多智能体协作的本质就是把这三个角色拆开。

单一对话模式

最基础的 AI 协作方式——你和一个 AI 助手在同一个对话窗口中完成所有任务。绝大多数人的起步模式。
✓ 核心优点
上手零门槛 — 打开对话框就能用，无需任何配置
上下文连贯 — 同一个窗口，前后关联自然衔接
简单任务效率最高 — 写一段PRD摘要、查一个技术概念，秒出结果
成本最低 — 单一对话，Token消耗最少
✕ 主要缺点
长对话"失忆" — 上下文越长，早期结论越模糊
所有任务串行 — 一个做完才能做下一个，无法并行
单一视角盲区 — 只有一个AI角色，缺乏多维度碰撞
上下文污染 — 市场分析的大段内容会干扰后续技术评估
核心使用建议
大多数任务的起点
但不应是终点
✓ 适合的场景
  • 单个功能点PRD：写一个"知识库文档上传"的需求描述
  • 竞品快查："腾讯IMA的HiChunk切分方案是什么？"
  • 技术概念理解："RAG和GraphRAG的区别是什么？"
  • 会议纪要整理：把一段评审录音转成结构化纪要
✕ 避免的场景
  • 新产品立项评估：需要市场+竞品+技术多维度交叉分析
  • 完整PRD撰写：功能定义+技术方案+测试标准一条龙
  • 售前方案设计：需要产品+算法+销售多角色碰撞
当你发现自己在一个对话里反复"提醒AI别忘了之前说的"——
是时候升级到多智能体模式了。

SubAgent 模式

Leader 分配独立任务给子 Agent，各自执行完毕后独立向 Leader 汇报。子 Agent 之间无任何沟通。
✓ 核心优点
上下文独立 — 每个子Agent只看自己的任务，不受干扰
启动成本低 — 无需定义角色Skill，Leader自动拆分派发
并行提速 — 多个子任务同时执行，总耗时≈最长子任务
Token消耗可控 — 子Agent用完即销毁，不持续占用上下文
✕ 主要缺点
完全隔离 — 子Agent之间无法沟通，互不知道对方在做什么
无法质疑补充 — 竞品发现的信号，技术评估无法针对性响应
Leader信息瓶颈 — 所有信息必须经Leader汇总，容易丢失细节
深度不足 — 简单汇总≠深度碰撞，产出缺乏多维度交叉验证
核心使用建议
适合"各干各的"
不需要商量的任务
✓ 适合的场景
  • 新品立项调研：竞品拆解、市场规模、技术可行性三线并行
  • 高校客户画像：同时分析双一流/普本/学科特色校三类客户群
  • 政策影响评估：教育数字化新政对产品线/销售/交付的独立影响
  • 季度经营分析：收入、续约率、客单价多指标独立拆解后汇总
✕ 避免的场景
  • 售前方案协作：销售和产品需要互相对齐客户需求才有深度
  • 产品开发全链路：PRD→算法方案→前后端→测试有严格依赖
  • 需要交叉验证：竞品发现的差异化点需要算法评估技术可行性
关键原则：SubAgent 是"分头干活"，不是"团队协作"。
任务之间越独立，SubAgent 效果越好。

Agent Teams 模式

同一任务交给 11 个角色（Leader + 10 位成员），成员之间可以互相商量、互相质疑、互相协作，最终由 Leader 批判性审视后综合决策。
✓ 核心优点
解决上下文瓶颈 — 每个成员上下文独立且聚焦
任务并行执行 — 多角色同步推进，效率翻倍
多维度协作 — 成员互相质疑，避免幻觉
贴合真实开发 — 完全复刻人类团队分工流程
灵活配置 — 自定义团队规模、角色，甚至不同成员指定不同模型
✕ 主要缺点
Token消耗巨大 — 每个成员独立实例，消耗是单Agent数倍
上下文冗余 — 每个成员重复加载项目基础上下文
文件冲突风险 — 任务拆分不清晰时，多成员修改同一文件
依赖分配要求高 — 设置不合理会变成"伪并行"
实验性功能 — 仍处于实验阶段，后续可能有重大变更
核心使用建议
不要为了换灯泡
召开董事会
✓ 适合的场景
  • 高教知识库立项评估：市场空间+竞品拆解+RAG技术方案+商业模式多维碰撞
  • 售前方案协作：销售带回客户需求→咨询拆解→产品出方案→算法评估可行性
  • Coral MVP 开发：产品经理出PRD→算法设计RAG→前后端评估工时→测试定标准
  • 学期经营复盘：市场分析师拆数据+竞品跟踪变化+Leader批判性审视出决策
✕ 避免的场景
  • 简单小任务：创建团队的开销大于收益
  • 串行强依赖：前后任务必须严格衔接
  • 同一文件多端修改：易造成代码冲突
关键原则：任务拆分一定要清晰，明确每个成员的职责和操作范围。
优先用于高价值、复杂的大型项目，让Token消耗的性价比最大化。

每个子Agent独立工作、独立汇报，彼此之间完全不知道对方在做什么。

🔒

05 · Agent Teams 交互演示

Agent Teams：协作开发 Coral MVP

高教版知识库 Coral 的 MVP 开发交给整个团队。Leader 拆出任务清单并指定负责人，
成员认领任务后互相商量、互相质疑、互相协作，最终 Leader 综合决策。

AGENT TEAMS MODE
任务清单 → 认领任务 → 多角色协作 → 综合决策
就绪
📌 下达任务
🔗
Agent Teams 特征：Leader 先拆出任务清单并指定负责人，成员认领后开始工作。过程中可以互相看到对方产出，直接发消息商量、质疑、补充——像一个真实团队协作开发。

10 · 团队优化路线

团队级优化路线

PHASE 1 · 已完成
角色定义 + Skill 编写 + 知识底座
11 个角色 Skill.md 全部完成，共享知识底座上线，自动归档和 Memory 机制就绪。每个角色可独立调用，也可通过 Leader 统一调度。
PHASE 2 · 进行中
实战验证 + 产出质量调优
用真实业务任务（立项评估、售前方案、Coral MVP）测试每个角色，根据产出质量迭代 Skill.md 的规则和模板。重点打磨 Leader 调度的稳定性和汇报质量——确保"带观点的决策"而非"搬运式汇总"。
PHASE 3 · 计划中
MCP 工具链集成
市场分析师接入 WebSearch 获取实时政策和招标信息；测试工程师对接 CI/CD 触发自动化测试；竞品调研员接入截图分析——从"纯文本推理"升级为"工具增强的实战分析"。
PHASE 4 · 计划中
专属知识注入
为每个角色添加专属知识文件：前端注入 UI 设计规范，后端注入系统架构文档，产品经理注入历史 PRD 库，算法总工注入成本计算模型，销售注入定价体系和中标案例。从"通用能力"进化为"讯飞高教专属能力"。
PHASE 5 · 远期
多场景协作流程固化
将高频协作场景（立项评审会、售前方案协作、季度经营分析、产品复盘会）固化为可一键触发的标准流程。每个场景预设参与角色、任务拆分规则和产出模板——从"每次手动指挥"进化为"一句话启动标准流程"。
