ONE-LINER · 一句话结论

GBrain 是 "个人 / 组织级 Agent 记忆系统的工程级参考实现"，我们的学科大脑是 "学科教学 / 科研场景的知识库产品"。底座架构高度同构——GBrain 几乎是我们"检索库 + 炼出库 + 行为积累库"三层架构的成熟工程版；但在搜索质量、知识治理、后台任务、引用防幻觉、技能化沉淀这五块，远走在我们前面，尤其后两块是 4-24 青林点出"治理短板"的现成答案。

CHAPTER 01 · REUSABLE

可直接复用的 8 项能力（拿来主义清单）

GBrain 的工程实现已经过 17,888 页生产数据验证。下列 8 项可直接借鉴架构、抄管道、套数据结构，省自研成本与试错周期。按建议优先级排序。
GBrain 模块 对应我们哪块 怎么用 优先级
混合搜索 5 层管道
向量 + BM25 + RRF + 图遍历 + 余弦重排		 R1 检索 + T1 问答 直接抄管道。我们当前只有"向量 + 关键字"两层，至少补 RRF 融合和反向链接推升——GBrain 的 BrainBench 实测：图谱开启时精度从 17.7% → 49.1%。 P0
零 LLM 自动链接
正则 + 类型推断从写入抽实体		 图谱库（炼出库） 红红方案靠 KAG 图谱构建——成本高、慢。加一层"零 LLM 链接"做底层兜底，KAG 只跑高价值实体，整体成本可降一个数量级。 P0
编译真理 + 时间线双结构
Compiled Truth 可重写 / Timeline 仅追加		 行为积累库 / 研究足迹 我们之前定义"研究足迹 = 何老师脉络 + 星火隐性知识 + 过程性存档"——GBrain 的 Compiled Truth + Timeline 是这个概念的现成数据结构。每个研究足迹页就用这个 schema。 P0
citation-fixer 引用修复 Skill
自动修复 + 来源追踪		 R3 论文写作 R3 写作刚需。论文场景必须解决"GPT 编引文"问题，直接抄 Skill 实现，作为论文场景上线前置门槛。 P0
小兵队列 Minions
Postgres 原生持久化后台任务		 库治理 / 长任务 性能对比：小兵 753ms / $0 / 100% 成功，子代理 >10s 超时 / $0.03 / 0% 成功率。备课批量、图谱治理、夜间增强——这些确定性后台任务不要走 SubAgent，走持久化队列。改我们的技术选型。 P1
三层 Enrichment
Tier1 / Tier2 / Tier3 分层丰富化		 炼出库的炼制流程 炼出库的术语 / 图谱 / 问答对，套用"重要实体走 Tier3 深度增强、长尾走 Tier1 轻量"的成本控制思路。 P1
soul-audit 身份审计
代理人设 / 边界 / 拒答规则		 五助（助教 / 助学 / 助研...） 五助每个角色都需要"人设 + 能力边界 + 拒答规则"三件套——GBrain 有现成 Skill 化模板。 P1
PGLite 单机嵌入式部署
30 分钟装好，无服务器		 学校私有化交付 高校私有化痛点是"运维重"。PGLite 单机模式可作为学院级试点的最小可用包，降低 POC 门槛。 P2
CHAPTER 02 · GAPS

我们方案没考虑到的 7 大缺口（必须补）

下列 7 项是 GBrain 已经实现、但我们 V5 方案完全或部分遗漏的环节。其中缺口 1、3、6 是4-24 场景收敛会议已被点过名的薄弱区，必须在 V6 方案里补齐。
GAP · 01
知识治理体系（21 个 cron 任务）
⚠️ 4-24 青林点过名
GBrain
跑着 19-21 个自主 cron：引用修复、孤立页检测、反向链接强制、健康检查、跨模态校验、技能验证、背链一致性、过期内容归档⋯⋯把"治理"做成了发条系统。
我们
只定义了"采集 → 治理 → 检索"三个动词，治理具体做什么、谁触发、多久跑一次——全部没定义。
怎么补
列我们自己的 cron 清单：术语去重 / 图谱孤点扫描 / 问答对失效检测 / 研究足迹归档 / 引用核验 / 跨学科冲突检测——画到 V6 库架构图，标注"治理层"。
GAP · 02
Skills 化的能力沉淀范式
范式缺失
GBrain
所有能力沉淀成 29 个 Skills，含 skill-creator + skillify + skillpack-check 三件套——能力可自我繁衍 + 自动健康检查。
我们
场景定义只到"输入示例 + 输出示例"，没有定义"这个场景怎么变成可复用的能力模块"，也没有质量校验机制。
怎么补
每个场景（T1 / T4 / R1...）落到 Skill schema（触发指令 / 5 步流程 / IO 契约 / 质量门槛 / 失败回退），且强制配套"质量校验 Skill"做兜底。
GAP · 03
幻觉防护 / 引用追踪机制
⚠️ 论文场景必备
GBrain
citation-fixer 列为常驻 Skill，所有写入必带来源追踪，所有引文必反查原文。
我们
R3 论文写作场景只写了"输出示例"，没有任何防幻觉 / 引文核验机制——这条若不补，论文场景上线即翻车。
怎么补
写作场景强制"每段输出 → 反查检索库 → 标注来源页"，引文未命中检索库直接拒答；上线门槛硬性指标：引文核验通过率 ≥ 95%。
GAP · 04
信号检测器（实时摄入不阻塞主流程）
行为积累库的引擎
GBrain
signal-detector 在用户与 Agent 对话时并行抽取实体 / 想法写入大脑——非阻塞、零感知。
我们
建库流程定义为"上传 → 解析 → 入库"——师生日常使用沉淀的"研究足迹"怎么自动产生，没说清。
怎么补
行为积累库需要一个"沉默观察者"：T1 问答 / R3 写作过程中自动抽信号，写入研究足迹页——这是"使用即建设"飞轮的真正引擎。
GAP · 05
评估基准（BrainBench）
效果证明缺失
GBrain
自己造了一个 240 页测试集，能跑出"图谱开 / 关"的精度差异：49.1% vs 17.7%——有数据说话。
我们
建库方案没有任何"怎么证明这个库有用"的基准——只能靠示意截图说服客户。
怎么补
每个场景定 20-50 条 ground truth 测试集，建库前 / 后对比；POC 时直接展示"我们的库 vs 通用大模型"的命中率差异。
GAP · 06
可读性优先 ≠ 数据库优先
⚠️ 高校学术自主权
GBrain
所有页面是 markdown 文件，数据库只是索引。承诺"代理可写、人类永远可改"——源代码永远可审计。
我们
默认是"数据库存储 + 前端展示"，老师改不了底层数据——这违背了高校用户的核心心理诉求。
怎么补
至少图谱库 / 术语库 / 研究足迹这三块支持老师"直接编辑底层 markdown"——这是高校用户的核心心理诉求（学术自主权 + 数据可带走）。
GAP · 07
跨模态融合（语音 / 邮件 / 会议 → 页面）
隐性知识源
GBrain
把会议转录、邮件、推文、语音通话都标准化成"页面"——四模态统一摄入。
我们
只考虑了"文献 / 课件 / 题目"结构化输入——录课视频、师生答疑录音、会议纪要这些隐性知识源全部漏掉。
怎么补
教师的录课视频、师生答疑录音、会议纪要——这些是高校最大的隐性知识源，方案里要补 media-ingest + meeting-ingestion 两条管道。
CHAPTER 03 · SKIP

不可复用 / 场景不匹配的部分

GBrain 服务的是"个人执行高管 / 研究者"的单租户场景，与我们"高校多租户 + 学科共享"的定位不同。下列 5 项不抄，但其中"多租户记忆边界"是我们比 GBrain 更难、也更值钱的差异化机会。
Twilio 语音电话接入
C 端单人场景，高校教学不需要电话与 Agent 交互。
Twitter / X API 摄入
国内场景不适用，对应可换成微信 / 公众号 / 知网，但优先级低。
个人执行高管场景（"为我准备 Jordan 会议"）
个人助理场景，与"五助"教学 / 科研定位不重合。
GStack 编码协作（设计决策 → 代码审查）
我们不做编码 Agent，跨产品桥接不必要。
⚡ 单用户人设强绑定（"你的"思想检索）
我们是多租户 + 学科共享模型，记忆的归属边界要重新设计——这是 GBrain 没解决的难题，是我们比 GBrain 更难、也更值钱的差异化机会。建议 V6 方案里专门一章。
CHAPTER 04 · ACTION

建议的下一步动作 · 3 段落地节奏

按短期 / 中期 / 长期分三段推进，重点是本周必须把"治理层" + "引用核验" + "Compiled Truth schema"补进 V6 方案，回应 4-24 会议未解决的问题。
SHORT · 本周
补 V6 方案三件套
  • 库架构图加一层 "治理 cron 清单"——直接回应 4-24 青林
  • R3 论文写作场景强制加 citation-fixer，否则不上线
  • 行为积累库页面 schema 直接抄 Compiled Truth + Timeline 双结构
MID · 下周
技术选型 + 评估基准
  • 各场景定义评估基准集（每场景 20-50 条 ground truth）
  • 技术选型评审：确定性后台任务走 Postgres 队列，不走 SubAgent
  • POC 模式确认：PGLite 单机包是否作为学院级试点交付物
LONG · 与红红讨论
差异化机会
  • 多租户记忆边界——GBrain 没解决，是我们的差异化机会，提到 V6 专门一章
  • 五助 × soul-audit 身份模板设计
  • 跨模态摄入（语音 / 视频 / 会议）的优先级排序
META INSIGHT · 元洞察

GBrain 给我们最大的启发不是"具体怎么做"，而是"知识库不是数据工程，而是 Agent 工程"——它是把所有能力沉淀成 Skills、把所有维护沉淀成 cron、把所有内容沉淀成可读 markdown 的"代理操作系统"。这个视角下，我们 V5 方案的"三层库"只是数据存储层，真正缺的是"这个库怎么活起来 + 怎么自我修复 + 怎么被代理操作"的运维层。V6 方案应该从"数据架构"升级为"Agent 运行时架构"。
