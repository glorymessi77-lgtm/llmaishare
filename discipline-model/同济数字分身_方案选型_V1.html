数字分身的 9 条技术路径，
为什么我们选RAG + Prompt而不是微调

同济数字分身方案的技术路径不是"唯一正确答案"，而是在 9 条可能路径中做的选择。 本文把所有方案盘点一遍：RAG 系列 / 多 Agent 编排 / 模型微调系列 / 知识图谱 / Hybrid 折衷—— 对每个方案做"技术路径 × 数据需求 × 工期 × 风险"四维评估，给出三期渐进推荐路径和 三个必须避开的坑。

方案 A · RAG + Prompt主推

用教师个人知识库 + 个性化 Prompt + 星火 X2，不做任何模型微调。快、可控、可扩展。

工期 · 29 人月 团队 · 7 人 风险 · 🟢 低

方案 F · 多 Agent 编排升级

在方案 A 基础上升级为 5 个 sub-agent 协作（风格/知识/引导/溯源/纠偏）。XClaw 最擅长的架构。

增量 · +5 人月 效果 · 85% 风险 · 🟡 中
PHASE 03 · 三期 · 明年

方案 I · Hybrid + LoRA选装

只对 3-5 位"镇店之宝"教师做 LoRA 精调。基于方案 F 加一层参数级个性化。

增量 · +8 人月 效果 · 92% 风险 · 🟡 中
01 · SCHEMES

9 条技术路径全景

每张卡标注推荐度（★1-5）、技术核心、工期估算、致命伤。5 星是一期主推，0 星是必须避开。
A
推荐度 · ★★★★★

RAG + Prompt Engineering

Retrieval-Augmented Generation + Prompt
向量检索 + 个性化 System Prompt + 星火 X2 生成。不做任何模型层面的个性化——教师的"灵魂"来自独立知识库命名空间 + Prompt 里的风格标签 + few-shot 示例。
数据教师现有语料即可（论文/视频/讲义）
基建Coral 现有 · 无额外投入
天花板70-80% 相似度
扩展性加教师 = 加知识库（1 小时）
工期29 人月 / 4.5 月 / 7 人团队
✓ 一期主推方案。Coral 基础设施现有，教师扩展到 100 位成本近似线性为零。风格模仿有天花板是它最大短板——但能在 8 月底交付的**唯一路径**。
B
推荐度 · ★★

LoRA 微调（每教师 adapter）

LoRA Per-Teacher Fine-Tuning
基础模型 + 每位教师一个 LoRA adapter（~10-100MB）。训练数据需求：每人 500-1000 对高质量问答对，外加风格语料。
数据每人 500-1000 问答对 · 稀缺
基建Coral E2 微调引擎 ○ 未具备
天花板85-90% 相似度
扩展性线性成本 · 20 位教师 = 20 倍
工期42 人月（含基建 6 + 评估 3）
✗ 不推荐作为主方案。致命伤三条：数据极难构造 + Coral 没有微调平台 + 扩展到 100 位教师成本爆炸。8 月底交付不完。可作为三期 Hybrid 选装包。
C
推荐度 · —

Full Fine-Tuning（完整模型）

Full Model Per-Teacher Training
每教师一个完整的微调模型（7B/14B 参数全量微调）。工程上不现实——列在这里只为堵住领导可能问"为什么不训练专属模型"。
数据每人万级问答对 · 不可达
基建每教师占一份 GPU 显存
天花板95% 相似度
扩展性完全不可扩展
工期100+ 人月
❌ 直接 pass。存储爆炸、算力爆炸、运维爆炸，20 位教师就要 20 份基础模型在线。只在研究场景有意义，B 端工程化无解。
D
推荐度 · ★★

P-Tuning · 轻量化 Prompt 训练

Prompt Tuning / P-Tuning v2
只训练一个 soft prompt 向量（几千到几万参数），不改基础模型权重。比 LoRA 更轻但效果也更有限。
数据每人 200-500 问答对
基建需微调平台 · 星火是否开放未知
天花板75% 相似度
扩展性线性成本 · 比 LoRA 轻 30%
工期35 人月
⚠ 不推荐。国内生态支持弱，星火 X2 是否开放底层接口未知。LoRA 搞不动的情况下，P-Tuning 也搞不动。
E
推荐度 · ★★★★

RAG + Agent Memory

RAG with Long-term Memory
方案 A 的增强版——加一层"长期记忆库"，把教师纠偏过的答案、学生常见问题存成可检索条目。本质是方案 A 的运营强化，不是独立架构。
数据运行时累积（教师纠偏）
基建Coral + 额外 KV 存储
天花板80% · 有"学习感"
扩展性⭐⭐⭐⭐ 与 A 相同
工期33 人月（比 A 多 3-4）
✓ 二期追加合适。一期 MVP 不强必要——先让方案 A 跑稳定，二期用它增强"越用越像"的感知。
F
推荐度 · ★★★★

Multi-Agent · 多智能体编排

Teacher Persona Agent Network
把"王教授数字分身"拆成 5 个 sub-agent 协作：🎭 风格 · 🧠 知识 · 🤔 苏格拉底 · 🔗 溯源 · ✏️ 纠偏。XClaw 作为 Agent Runtime 天然适配这种架构。
数据同 A · 无额外要求
基建XClaw 天然支持
天花板85% · Wow 感最强
扩展性⭐⭐⭐⭐ 高
工期34 人月（+5 vs A）
✓ 二期升级首选。"竞对是单个 AI 在装教师，我们是小团队在协同装教师"——这是 XClaw 作为 Agent Runtime 的差异化护城河。
G
推荐度 · ★★

Knowledge Graph-Centric

Pure Graph-Based Reasoning
完全靠教师的学科知识图谱做推理回答，不依赖向量检索，不依赖 LLM 生成风格。适合强结构化学科，不适合人文社科。
数据图谱 · 手工构建 · 极难
基建Coral M2 仅 Excel 导入 · 严重不足
天花板60%（通用场景）
扩展性⭐⭐ 学科覆盖窄
工期45+ 人月
⚠ 不推荐。同济建筑史/哲学等王牌文科无法图谱化。可作为方案 A 的"专业学科增强模块"——比如高数数字分身叠加图谱推理层。
H
推荐度 · ★★

DSPy · 可训练 Prompt

Programmable Prompt Framework
用 Stanford 的 DSPy 框架把 Prompt 变成"可训练的对象"——教师纠偏会自动反向训练 Prompt。技术前沿但风险高。
数据A 的数据 + 纠偏数据

天花板85%
扩展性⭐⭐⭐ 有一定扩展
工期37 人月（+学习曲线）
⚠ 不推荐作为主方案。国内 B 端几乎没人用过，标杆项目用前沿技术是赌博。可作为 F2 纠偏 Skill 的底层技术选型——但这是二三期的事。
I
推荐度 · ★★★★

Hybrid · RAG + LoRA 组合

RAG for All + LoRA for VIPs
大部分教师走方案 A（RAG），只对 3-5 位"镇店之宝"教师做 LoRA 精调。最务实的折衷方案，分期交付友好。

03 · PARADIGM

技术实现差异的本质 · 4 大范式

9 个方案看似很多，其实分成 4 大范式。理解范式差异比记住 9 个名字更重要。
01

知识检索 + 提示工程

方案 A / E / H

模型不变，改"喂给模型什么"。教师的个性来自 Prompt 上下文和检索到的材料。迭代快、数据门槛低，是 B 端首选。

02

多智能体编排

方案 F

模型不变，改"怎么调度多个模型实例"。把单个 Prompt 拆成多个 sub-agent 协作。XClaw 作为 Agent Runtime 最擅长。

03

参数级个性化

方案 B / C / D

改模型本身的参数。技术最深、效果最到位，但数据门槛高、扩展性差、基建缺。B 端不适合走这条路。

04

图谱符号推理 / 混合

方案 G / I

绕开 LLM 用确定性推理（G），或范式 1 兜底叠加范式 3 加 buff（I）。适合特定场景，不是主路径。

05 · PITFALLS

三个必须避开的坑

不是小瑕疵，是会让项目直接塌方的硬伤。
PITFALL · 01

把数字分身等同于"模型个性化"

最容易踩的认知陷阱。领导听到"数字分身"会联想"训练专属模型"——但那条路（B/C/D）在数据稀缺 + 基建缺失 + 扩展性差三方面都不通。数字分身 = 知识库个性化 + Prompt 个性化，不是模型个性化。

PITFALL · 02

为 1% 的效果增益付 50% 的工期代价

LoRA 比 RAG 的效果从 80% 升到 90%，但工期从 29 增到 42。每 1% 效果提升要 1.3 人月——对于 8 月底必须交付的标杆项目，这个代价不划算。要追求效果，走二期的多 Agent 升级（+5 人月 +5% 效果）性价比高得多。

PITFALL · 03

盲目追求"技术最先进"

DSPy（方案 H）是最前沿的，但团队没用过、国内没生态。在标杆项目里用前沿技术是赌博——万一卡壳郑书记不会听解释。标杆项目的第一要义是不能翻车，技术深度让位于交付确定性。
